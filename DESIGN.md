## 1. 产品定位与核心实验论点

**一句话定位**："AI不持续盯着摄像头，只在用户主动说话时抓取一次当前画面，并通过成本面板可视化'看/听'的开销。"

**核心实验闭环**：按住 🎤 说话 → 松手，语音转文字填入输入框 + 截取当前帧暂存 → 点发送 → 文字 + 图像一起发送至 Qwen-VL → 文字回复 + TTS 播报 → 成本面板更新。

**核心论点**：在用户低频提问场景下，"按需发送图像"比"持续发送（1fps）"可节省 90%+ 的图像调用成本，同时保持相同的对话体验质量。

## 3. 成本控制技巧：想到的 vs 实际采用的

### 想到的

| 技巧 | 原理 | 采纳？ |
|------|------|--------|
| 帧时间间隔节流 | 至少间隔 N 秒才发送一次图像 | ✅ D2 采纳 |
| 像素变化检测 | 只有画面有明显变化才发送 | ✅ D2 采纳 |
| 多帧采样+最佳帧选择 | 说话期间采样队列，选变化最大的帧 | ✅ D2 采纳 |
| 图像压缩 | 前端 Canvas 缩放至 320px + JPEG 质量 0.5 | ✅ D1 采纳 |
| 服务端帧去重 | 哈希缓存去重（5min TTL） | ❌ 移到前端 |
| 预算告警 | API 调用次数上限预警 | ❌ 不做 |

### Token 估算公式（来源：阿里云/Anthropic 官方文档）

- **图像 token** ≈ `Math.ceil(width × height / 750)`（Anthropic 官方近似公式，README 中声明此公式来源）
- **文本 token** ≈ `Math.ceil(text.length / 1.5)`（中文粗略估算）

## 4. 权衡与放弃方案

| 方案 | 放弃理由 |
|------|---------|
| VAD（语音活动检测） | 浏览器原生不支持可靠 VAD；PTT（按住说话）模式用户感知明确，已足够 |
| 持续视频流（WebRTC） | 与核心论点冲突——我们要证明"不需要持续发" |
| 云端 ASR | 增加延迟和成本；浏览器原生 Web Speech API 免费且延迟低 |
| 多模型切换 | Demo 范围外，用一个模型（qwen-vl-max）即可 |
| 数据库持久化 | Demo 场景；刷新重置，符合预期的临时交互 |
| 用户登录/认证 | Demo 场景；无用户体系，成本极低无需防盗用 |

## 5. 实验设计

### 实验基准说明（Baseline Assumption）

本实验对比的"方案A"与"方案B"均指**图像数据的发送策略**：

- **方案A（持续发送）**：模拟连接建立期间，以 1fps 持续向云端发送图像帧的理想情况（参考业界持续感知型实时多模态方案，如 Gemini Live API / WebRTC 持续帧传输）。

- **方案B（按需发送，本方案）**：完全事件驱动——仅当用户主动发起一次语音提问时，才从端侧采样队列中挑选一帧发送。

两者的核心差异**不是"方案A刻意设计得低效"**，而是反映了两种真实存在的产品范式："持续感知"范式 vs "按需感知"范式。本实验的目的正是量化这两种范式在"用户低频提问"场景下的成本差异。

### 对比方法

- 方案A：以 `elapsedSeconds × 1fps` 计算发送帧数，每帧 token = `estimateImageTokens(320, 240)`
- 方案B：以实际 `imagesSentCount` 计算
- 节省比例 = `(planA.cost - planB.cost) / planA.cost × 100%`

#### 成本对比

| 方案 | 单次图片数 | 总图片数 | 估算图片Token | 估算费用 |
|------|-----------|---------|-------------|---------|
| **方案A**（持续发送 1fps） | ~267 张 | ~801 张 | ~82,467 | ~$0.034 |
| **方案B**（本方案实测） | 1 张 | 3 张 | 309 | ~$0.00013 |

- **图片发送节省率**: **~99.6%**
- **费用节省**: 约 260 倍
- 定价基准：Qwen-VL-Max 输入 $0.41/1M tokens，输出 $1.23/1M tokens
